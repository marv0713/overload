# 多出口插件化分发架构 (Pluggable Publisher Architecture)

既然你希望做到**“一个文章可以有多个出口（发邮件、推微信、存草稿箱），且每个播客源可以独立配置”**，并且我们已经有了 PushPlus 接口，我们需要对目前的推送逻辑进行一次解耦重构。

目前的痛点是：`process_sources.py` 把“推微信草稿”的逻辑写死了，只要开启推送，所有跑出来的文章全塞进草稿箱，且仅仅通过 PushPlus 发一个“提示消息”。

## 改造方案

我们将引入一个真正的**“插件化发布器 (Publishers)”**架构。

### 1. 配置层升级 (`SourceConfig`)
给每个订阅源新增一个 `destinations` (发布目的地) 数组字段：
```json
{
  "name": "再市为人",
  "writer_profile": "dehydrate",
  "destinations": ["pushplus"]  // 只推给个人微信
},
{
  "name": "Unrivaled Investing",
  "writer_profile": "deep-stock-analysis",
  "destinations": ["wechat_draft", "pushplus"] // 既存公众号草稿，又发微信通知
}
```

### 2. 代码解耦 (`src/youtube_to_wechat/publishers.py`)
新建一个独立的 `publishers.py` 模块，使用插件模式（Strategy Pattern）定义各种出口：
*   **`WechatDraftPublisher`**: 原封不动保留现有推公众号草稿箱的逻辑。
*   **`PushPlusPublisher`**: 专门负责把文章的 **正文内容（Markdown）** 通过 PushPlus 发送到你的个人微信（而不是只发个提示）。由于微信对长度有限制，过长的内容可以考虑发全文或截断。
*   *(未来扩展)* `EmailPublisher`, `ArchivePublisher` 等。

### 3. 主流程重构 (`scripts/process_sources.py`)
把原来从 530 行开始又长又臭的写死逻辑全部砍掉，换成优雅的遍历调用：
```python
if args.push:
    for dest in candidate.source.destinations:
        publisher = get_publisher(dest)
        publisher.publish(article_path, cover_path, env)
```
同时将命令行参数 `--push-draft` 更名为更通用的 `--push`。

### 4. 数据更新
去 Supabase 数据库里，把“再市为人”这个频道的配置更新一下，加上 `"destinations": ["pushplus"]`；把其他的频道加上 `"destinations": ["wechat_draft"]` 作为默认。
